# Decision Log: Pick/Pack Flow
## Session 002 ŌĆö March 28, 2026

### Context
Mode 3 (PRD Generation) and Mode 4 (User Story Generation) executed in parallel for Pick/Pack feature. Inputs: Mode 1 competitor research (March 18, 2026), Mode 2 flow analysis (March 23, 2026), and the combined MVP recommendation (Option 3 Smart Ship Wizard + Option 2 Persistent Tracking).

---

### Decision 1: MVP Scope Confirmation
**Question:** Which flow option forms the basis for the PRD and user stories?

**Decision:** Combined MVP ŌĆö Option 3 Wizard + Option 2 Persistent Tracking, as recommended in the Mode 2 flow analysis.

**Rationale:**
- Wizard provides fastest fulfillment path for solo operators (3-10 person shops)
- Persistent PickList records + FulfillmentStatus close the visibility and interruption gaps that standalone Option 3 had
- Combined approach scored highest on decision criteria (UX heuristic 30, strong operational reality for both small and large teams)
- Consistent with existing prototype patterns (870px modal, table sections)

**Impact:** All PRD requirements and user stories are built around this combined approach. The three user paths (all-in-one, save & resume, bypass) are first-class flows.

---

### Decision 2: Data Model Approach
**Question:** How to extend the existing type system for pick/pack?

**Decision:** New types added:
- `FulfillmentStatus` enum: "Not Started" | "Pick in Progress" | "Picked" | "Packed" | "Shipped"
- `PickList` interface: first-class entity tied to SalesOrder
- `PickListItem` interface: per-item pick tracking with status
- `PackingSlip` interface: generated document record
- `fulfillmentStatus` field added to `SalesOrder` and `SalesOrderSummary`

**Rationale:**
- FulfillmentStatus as a separate field from OrderStatus (non-breaking ŌĆö existing status logic untouched)
- PickList as a first-class entity enables reprinting, auditing, and resume capability
- Follows existing patterns (same approach as ShippedItem records)

**Impact:** Moderate data model expansion. New types.ts interfaces. SalesOrderSummary gains a column on the list screen.

---

### Decision 3: User Story Categorization
**Question:** How to organize the 44 user stories for implementation planning?

**Decision:** 8 operations-led categories:
1. Core Operational Flow (9 stories) ŌĆö happy path
2. Exception & Error Handling (5 stories) ŌĆö edge cases
3. Role & Permission Behavior (4 stories) ŌĆö multi-user
4. State Communication & Audit (6 stories) ŌĆö visibility
5. Recovery & Edit Behavior (5 stories) ŌĆö undo flows
6. Bulk & High-Volume Usage (3 stories) ŌĆö power users
7. Configuration & Admin (3 stories) ŌĆö settings
8. Ecosystem Connections (4 stories) ŌĆö integrations

**Rationale:** Categories reflect how real businesses use the product ŌĆö not just the happy path. Prioritized as Must Have (core + state + recovery), Should Have (exceptions + roles + config), Nice to Have (bulk + ecosystem Phase 2 items).

**Impact:** Implementation roadmap: ~4-5 weeks for Must Have stories, 2-3 weeks for Should Have, deferred items for Phase 2.
