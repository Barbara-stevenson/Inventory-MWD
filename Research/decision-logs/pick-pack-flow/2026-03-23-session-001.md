# Decision Log: Pick/Pack Flow
## Session 001 — March 23, 2026

### Context
Mode 2 (Flow Analysis & Options) executed for Pick/Pack feature. Three distinct flow options designed based on competitor research across 7 competitors and internal data from 465 customer transcripts. Predecessor: Mode 1 competitor research completed March 18, 2026.

---

### Decision 1: Flow Architecture Approach
**Question:** How should pick/pack integrate with the existing Sales Order screen and shipping flow?

**Options presented:**
1. **Fulfillment Sidebar** — Collapsible side panel on the SO screen for inline pick/pack (inspired by inFlow, SOS)
2. **Staged Fulfillment** — Separate Pick → Pack → Ship stages with independent status tracking (inspired by Cin7, Fishbowl)
3. **Smart Ship Wizard** — Multi-step wizard within the existing ship modal (inspired by Katana, inFlow)

**Agent recommendation:** Option 3 (Smart Ship Wizard) for MVP, with Option 2 elements as Phase 2

**Rationale:**
- Lowest implementation risk — extends existing ship modal rather than building new infrastructure
- Passes the "2-minute test" for non-specialist operators
- Matches Method's current customer base (3-15 person operations)
- Preserves fast path (Quick Ship All) for simple orders
- Aligns with all MVP design constraints (manual-first, no serial tracking, no barcode)
- Option 2's FulfillmentStatus is the best long-term architecture but adds scope that conflicts with MVP goals

**Decision:** Pending user review — options presented for selection

**Impact:** This decision shapes the PRD (Mode 3), user stories (Mode 4), and prototype implementation (Mode 7). Choosing Option 3 means minimal type changes and no new SO screen sections; choosing Option 2 means new PickList entity, FulfillmentStatus type, and list screen changes.

### Decision 2: Packing Slip Generation Pattern
**Question:** When and how should packing slips be generated?

**Options across the three flows:**
1. Inline generation from sidebar (Option 1)
2. Part of formal Pack stage (Option 2)
3. Generated at ship confirmation within wizard (Option 3)

**All options include:**
- Price visibility toggle (Phil Helms requirement)
- PDF generation for printing
- Re-printable after the fact

**Decision:** Pending — tied to flow architecture decision above

---

### Decision 3: MVP Status Model
**Question:** Should new fulfillment states be added to the data model?

**Options:**
1. No new states — pick/pack happens inside modal, only OrderStatus changes at ship time (Option 3)
2. New FulfillmentStatus enum on SalesOrder with 6 states (Option 2)

**Agent recommendation:** No new states for MVP (Option 3 approach). Add FulfillmentStatus in Phase 2 based on real user feedback.

**Decision:** Pending user review
