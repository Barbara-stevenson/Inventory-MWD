Step 11: Summary — Pick & Pack (2 Steps) + Ship

The Three Tables

Section What it shows Items move here when... Items leave when...
Items not shipped Order line items not yet in fulfillment Order is created User picks them (via Pick & Pack modal)
Pick list Items being fulfilled (picking, picked, packed) User adds them via Pick & Pack modal User ships them (via Ship modal)
Items shipped Completed shipments User ships packed items User clicks Undo (reversible)

The Three Modals

Modal Triggered by Shows Does
Pick "Pick & Pack" button Items from "Items not shipped" Moves items to Pick List, prints pick list
Pack [Pack ▸] button Picked but not yet packed items Marks as packed, generates packing slip
Ship "Ship ∨" button Packed items from Pick List Moves items to "Items shipped"

Key Principle

Each modal only shows items in the right state. You can't pack what isn't picked. You can't ship what isn't packed. Items move through the tables as they progress.
This walkthrough demonstrates the non-linear, flexible nature of the Pick & Pack wizard (2 steps: Pick → Pack) plus Ship action. Teams can work at their own pace, handling items independently. Widget A can ship while Bracket B is still being packed and Motor D is still being picked. The three tables keep everything organized, and the modals handle all the transitions.

Unpack Flow

Packed items in the Pick List can be reverted to "picked" status via an Unpack action (accessible through more options or a dedicated button). FulfillmentStatus changes: Packed → Picked. This is useful if packing was done incorrectly or items need to be repacked differently.

Note: Unpacking doesn't move items back to "Items not shipped" — it just changes their status from Packed back to Picked. They remain in the Pick List for potential re-packing or shipping.

Undo Ship (Existing Pattern — FR-308)

Items in the "Items shipped" table can be moved back to "Items not shipped" via an Undo button on the Items Shipped table header. This is an existing pattern in Method that reverses a shipment action.

Clicking Undo moves Widget A back to the Pick List with status "Picked." This allows the user to re-pack, re-address, or change shipping carrier before re-shipping. The shipment is reversed, and FulfillmentStatus reverts from "Shipped" to "Picked."

The three undo/reverse actions work together to provide full flexibility. Unpick moves items back to the order queue, Unpack changes status without moving items, and Undo Ship brings shipped items back to the Pick List. This gives users complete control over the fulfillment process and allows them to correct mistakes or change fulfillment strategy mid-stream.

Method's advantage: Everything stays on the Sales Order screen. No separate modules, no mobile app required, no WMS. Three modals handle the entire flow. The existing Ship pattern is preserved — Ship ∨ works the same way, it just now only shows packed items. This matches Method's target users: 3-50 person operations that need structure without a separate warehouse system.
What Method borrows from competitors:
  • From Fishbowl: Color-coded availability (🟢🟡🔴) in the Pick modal
  • From Katana: "Pack some" as an explicit action (the [Pack ▸] button)
  • From Cin7: Per-line-item status tracking through stages
  • From inFlow: Everything on the order screen, no separate modules
Method's Pick & Pack wizard (2 steps) plus Ship action is competitive and practical. It combines the best practices from Fishbowl, Katana, Cin7, and inFlow while staying true to Method's design philosophy: everything on the order screen, no separate systems, built for small-to-mid market teams.
