MVP Recommendation

Combined Approach

Pick & Pack Wizard + Separate Ship Action + Option 2 Persistent Tracking

The MVP is a 2-step Pick & Pack wizard (combined with a separate Ship dropdown) paired with Option 2's persistent state tracking (fulfillment infrastructure).

Why combine them? The wizard is the fastest fulfillment interaction. Separating Ship from Pick & Pack lets users choose when to ship after picking and packing. Without persistent tracking, the team can't see fulfillment progress, audit what was picked, or resume mid-flow. Option 2's infrastructure closes these gaps.

What the MVP Includes

1. Pick & Pack Wizard (2-Step Modal)

The 2-step modal flow (Pick → Pack) as the primary fulfillment interaction. Ship is a separate action via the "Ship ∨" dropdown button. "Ship All" preserved as fast path.

2. FulfillmentStatus on List (from Option 2)

New column on Sales Orders list. Values: Not Started → Pick in Progress → Picked → Packed → Shipped. Answers "which orders need picking?"

3. Persistent PickList Records (from Option 2)

When user completes Step 1, a PickList record is saved. Enables: reprinting, auditing, resuming if wizard closed mid-flow.

4. Pick List Section (from Option 2)

New section on SO detail screen. Shows pick list status, items picked, print button. Includes Pack and Unpick actions. Matches existing table pattern.

Undo / Reverse Flow (Unpick & Unpack)

Competitor undo support: Every major competitor supports undo at pick/pack stage. inFlow has "Unfulfill" button + "Unpick all" on mobile (first-class undo). Fishbowl uses CTRL+D void. Katana allows reverting to "Not delivered" status. Cin7 supports undo sale to Draft, which reverses the entire fulfillment chain. Critical rule: undo is always available before shipping.

Undo Paths — Three Routes to Reverse Fulfillment

UNPICK (Pick List → Items not shipped):
  Pick List section → [Unpick ▸] button on section header
  Opens Unpick modal → select items → moves back UP to Items not shipped
  FulfillmentStatus reverts: Picked → Pick in Progress → Not Started

UNPACK (Pick List "packed" → Pick List "picked"):
  Unpack option in Pack modal or contextual action
  Packed items revert to "Picked" status
  FulfillmentStatus reverts: Packed → Picked

NO UNDO AFTER SHIP:
  Existing Undo button on Items Shipped table (FR-308)
  Moves items back to "Items not shipped"
  Removes from ShippedItem table
Key points: Unpick works only before the pack stage. Unpack works only before ship. After shipment, the existing Undo button on the Items Shipped table handles reversals (different flow).

Option 1: Fulfillment Sidebar

Embedded Panel

Philosophy

Pick and pack are contextual actions on a sales order, not separate workflows. A collapsible sidebar panel provides picking instructions, pack confirmation, and packing slip generation inline on the SO screen.

Why this approach: Inspired by inFlow's tab-based approach and SOS Inventory's model. Method's 3–50 person customers often have the same person creating the SO, pulling items, packing, and shipping. A separate module adds friction for small teams.

Pros

  • Zero navigation — everything on the SO screen
  • Reuses existing shipping infrastructure
  • Optional picking — skip to "Ship All"
  • Pick list and packing slip are SO views, not separate entities
  • Minimal new types needed
  • Lowest implementation complexity

Cons

  • Pick/pack status not tracked independently
  • Sidebar is new to the prototype (consistency)
  • May be slow for power users (50+ orders/day)
Best for: The office admin at a 5-person shop who creates SOs, walks to the shelf, pulls items, packs, and ships — all in one sitting.

Option 2: Staged Fulfillment

Sequential Workflow

Philosophy

Picking, packing, and shipping are operationally distinct activities that may be done by different people at different times. The system tracks each stage independently. This is a structured, stage-gated workflow.

Why this approach: Draws from Cin7's 3-stage sequential model (Pick → Pack → Ship) and Fishbowl's status progression. For Method's larger customers (20–50 person warehouses), the person creating the SO is NOT the person picking items. The flow needs handoff points between roles.

Pros

  • Full visibility into fulfillment progress
  • Supports role separation (office → picker → packer → shipper)
  • Pick list is referenceable, printable, auditable
  • Availability indicators prevent wasted picks
  • Stage gates prevent fulfillment errors
  • Consistent with existing modal patterns
  • FulfillmentStatus enables queue management

Cons

  • More clicks to complete fulfillment (4+ actions)
  • FulfillmentStatus adds complexity
  • Pick list as first-class entity = more data to maintain
  • Overkill for small teams
  • Stage gates can feel bureaucratic for simple orders
Best for: The 20–50 person warehouse with dedicated pickers and packers who need handoff points and visibility into process.

Option 3: Pick & Pack Wizard

2-Step Modal Flow

Philosophy

Pick and pack should feel like a natural, focused workflow. A 2-step wizard within the existing modal pattern guides the user through pick → pack. Ship is a separate action, available when ready.

Why this approach: Separates picking/packing from shipping, giving the user more control over timing. A 2-step wizard within a single modal keeps the fulfillment process focused and fast, while the Ship dropdown remains available for later.

Pros

  • Fewest changes to existing architecture
  • Familiar modal pattern (users know SelectShippedItemsModal)
  • Pick list and packing slip generated as part of flow
  • Step indicator provides structure without rigidity
  • "Ship All" dropdown preserves fast path
  • Carrier/tracking fields in Ship action
  • Experienced user: pick → pack in 2 minutes, ship when ready

Cons

  • No persistent pick list record
  • No fulfillment status on list screen
  • Wizard modal is new UI pattern
  • Doesn't support role separation well
  • Modal state may be lost if closed
  • No batch picking across orders
Best for: The 3–10 person operation where one or two people handle everything from SO creation to shipping. They want structure without bureaucracy.
