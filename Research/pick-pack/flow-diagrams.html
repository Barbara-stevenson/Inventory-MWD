Option 1: Fulfillment Sidebar

Alternative Score: 24/35
Philosophy: Pick and pack are contextual actions on a sales order, not separate workflows. A collapsible sidebar panel provides picking instructions, pack confirmation, and packing slip generation — all without leaving the SO screen. Inspired by inFlow's tab approach and SOS Inventory's model where the shipment record IS the packing slip.

✅ Pros

  • Zero navigation — everything on the SO screen
  • Reuses existing shipping infrastructure
  • Optional picking — skip to Ship All for simple orders
  • Lowest implementation complexity
  • Pick list + packing slip are views of SO data (no new entities)

❌ Cons

  • Sidebar is a new UI pattern (prototype uses modals)
  • No batch picking across multiple orders
  • No independent pick/pack status tracking
  • May feel cramped on smaller screens
  • Doesn't support role separation (picker ≠ shipper)

Option 2: Staged Fulfillment

Highest Score Score: 28/35
Philosophy: Picking, packing, and shipping are operationally distinct activities that may be done by different people at different times. Each stage is tracked independently so the team always knows where every order stands. Inspired by Cin7's 3-stage model and Fishbowl's status progression with color-coded availability.

✅ Pros

  • Full visibility — FulfillmentStatus on list screen
  • Supports role separation (office → picker → packer → shipper)
  • Pick list is a referenceable, auditable document
  • Availability indicators (🟢🟡🔴) prevent wasted picks
  • Stage gates prevent shipping un-picked items
  • Consistent with existing modal patterns

❌ Cons

  • Most clicks to complete (4 distinct interactions)
  • New FulfillmentStatus adds data model complexity
  • New PickList entity requires CRUD + state management
  • Higher implementation scope for MVP
  • Power users (50+ orders/day) may find it ceremonial
  • Risk of over-engineering per research warning

Option 3: Smart Ship Wizard

✓ Recommended for MVP Score: 26/35
Philosophy: Don't add new stages — enhance the existing shipping flow to include picking confirmation and packing slip generation as optional enrichments within a single continuous flow. A 3-step wizard inside the existing ship modal handles pick → pack → ship in one sitting. Inspired by Katana's streamlined approach and inFlow's all-in-one model.

✅ Pros

  • Fewest changes to existing architecture
  • Familiar modal pattern (users know SelectShippedItemsModal)
  • Step indicator provides structure without rigidity
  • "Quick Ship All" preserves fast path
  • Carrier + tracking fields added naturally
  • Passes the 2-minute fulfillment test
  • Minimal data model impact — no new types

❌ Cons

  • No persistent pick list record (can't audit later)
  • No fulfillment status on list screen
  • Wizard is a new pattern (existing modals are single-step)
  • Doesn't support role separation well
  • Modal state lost if closed accidentally
  • Packing slip only generated at ship time

Recommendation

Option 3 (Smart Ship Wizard) for MVP — lowest implementation risk, extends existing ship modal, passes the 2-minute fulfillment test, matches Method's current 3–15 person customer base. Then layer in Option 2's FulfillmentStatus tracking in Phase 2 based on real user feedback.
